# Policy: Configuration Documentation

## Purpose

Keep source-controlled configuration discoverable and prevent drift between the
actual `config/` tree and the human-readable configuration reference.

## Rule

Every source-controlled configuration asset under `config/` must be reflected in
`docs/CONFIGURATION_REFERENCE.md`.

The generated compact user-facing companion doc is
`docs/USER_SETTINGS_REFERENCE.md`. It must stay bucketed by stable user-facing
categories, and generation must fail if a persisted setting is not assigned to
one of those buckets.

Both docs are generated by
`tools/governance/generate_configuration_reference.py` and must not be edited by
hand.

## Scope

This policy applies to:
1. runtime and governance config assets under `config/`
2. keybinding, menu, gameplay, playbot, topology, tutorial, audio, and project
   config files
3. non-JSON config assets under `config/` that affect tooling or governance
4. excludes `config/schema/`, which documents validation contracts rather than
   runtime parameter values

## Required workflow

Use this order:
1. change the canonical config source under `config/`
2. regenerate `docs/CONFIGURATION_REFERENCE.md` and `docs/USER_SETTINGS_REFERENCE.md`
3. run `tools/governance/generate_configuration_reference.py --check` or
   `./scripts/verify.sh`
4. update relevant RDS/backlog/current-state docs when behavior or governance
   changed

## Exceptions

Exceptions are allowed only when documented and one of these applies:
1. the file is a schema contract under `config/schema/`
2. the file is generated state under `state/` rather than source-controlled
   config
3. the asset is binary and needs a companion human-readable explanation instead
   of inline parameter listing

## Review and enforcement

1. Config changes that alter generated configuration docs must include regenerated
   `docs/CONFIGURATION_REFERENCE.md` and, when relevant,
   `docs/USER_SETTINGS_REFERENCE.md` in the same batch.
2. `./scripts/verify.sh` must fail when either generated configuration doc is stale.
3. `docs/USER_SETTINGS_REFERENCE.md` bucket structure and label resolution must come from canonical runtime/config sources, not ad hoc literals.
4. Keybinding profile summaries inside `docs/USER_SETTINGS_REFERENCE.md` must derive category labels/descriptions and scope ordering from canonical keybinding category docs, not from hardcoded profile-shape assumptions.
5. Governance manifests and policy docs must stay synchronized with this rule.
